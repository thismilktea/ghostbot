# GhostBot 测评体系说明（重写版）

## 1. 先用一句话说清：GhostBot 到底在测什么

GhostBot 现在的主测评体系，测的是：

> **给 Agent 一个固定起点的项目副本，再给它一条或多条用户要求，观察它最终是否在约束内完成任务，以及它完成任务的过程质量如何。**

这里有四个关键词：

1. **固定起点**：每个场景都从同样的项目初始状态开始。
2. **明确要求**：不是泛泛问答，而是带目标、范围、验证要求的任务。
3. **约束内完成**：不仅要“做成”，还要不越界、不乱改、不超预算。
4. **过程质量**：即使都做成了，也要区分谁更快、谁更省 token、谁工具抖动更少、谁更会自我修正。

所以它不是在测“模型会不会聊天”，而是在测：

- 能不能在代码项目里完成任务
- 完成得对不对
- 完成得稳不稳
- 完成得经济不经济

---

## 2. 一个评测样本，最小长什么样

可以把每个评测样本理解成一条这样的记录：

### 2.1 起点
给 Agent 一个项目目录，或者一个空目录。

例如：
- 一个小型商店项目
- 一个空目录 scaffold 场景

### 2.2 用户要求
给它一条任务说明，例如：

- “修复购物车对负数数量缺少校验的问题，并运行 pytest tests/test_cart.py 验证。”
- “为购物车数量边界补充测试，只修改测试文件，并运行 pytest tests/test_cart.py。”
- “请阅读这个项目，给出一个简短优化计划，不要修改代码，并在回复中包含优先级、验证、风险。”

### 2.3 约束条件
同时定义这条任务的边界，例如：

- 必须改哪些文件
- 禁止改哪些文件
- 最多允许改几个文件
- 最多允许调用多少次工具
- 必须运行哪些检查命令
- 回复里必须包含哪些词

### 2.4 输出结果
Agent 跑完以后，系统记录：

- 最终回复
- 改了哪些文件
- 调了多少次工具
- 哪些工具重复调用了
- 外部检查命令过没过
- 总耗时、总 token
- 是否中途报错
- 是否出现“失败后又修回来”的行为

然后基于这些结果打分和判定。

---

## 3. 一次评测的完整流程

一条评测场景从头到尾，大致是下面这个流程：

### 第 1 步：复制一个干净起点

每个场景先准备一个独立工作目录。

意思是：
- 如果这是商店项目场景，就复制一份商店项目
- 如果这是空目录场景，就新建一个空目录

这样每次评测都从同样起点开始，结果才可比较。

### 第 2 步：把任务交给 Agent

系统把场景中的 prompt 发给 GhostBot。

此时 Agent 会像平时一样：
- 读文件
- 搜索代码
- 编辑文件
- 运行测试
- 生成回复

### 第 3 步：记录执行轨迹

系统会记录整个过程里的关键行为：

- 每轮迭代
- 每次工具调用
- 工具调用是否报错
- 最终停下来的原因
- token usage

这一步的意义是：

> 不只是看最后结果，还能分析“它是怎么做到的”。

### 第 4 步：比对文件变化

任务结束后，系统把“开始前的目录快照”和“结束后的目录快照”做比较，得到：

- 改动文件列表
- 新增文件列表
- 删除文件列表

### 第 5 步：执行外部验证命令

如果这个场景要求跑检查，例如：

- `python -m pytest tests/test_cart.py`

系统就真的在该场景工作目录里执行这条命令，并记录：

- exit code
- stdout
- stderr
- 是否通过

### 第 6 步：做硬性判定

系统接着检查：

- 有没有改到必须改的文件
- 有没有误改禁止改的文件
- 有没有超出允许改动范围
- 有没有超过工具调用上限
- 外部检查是否通过
- 回复里有没有包含要求的关键短语

这些都属于**硬规则**。

### 第 7 步：计算行为指标

除了 pass/fail，还会把过程统计成一组指标，例如：

- latency
- tokens
- tool calls
- redundant tool rate
- requirement hit rate
- self-correction rate
- recovery cost
- check pass rate

### 第 8 步：可选地做 LLM judge

如果该场景配置了 judge，就让模型基于：

- 原始任务
- 验收条件
- 实际改动文件
- 外部检查结果
- 最终回复

给出：
- score
- passed / failed
- verdict
- reasoning

但注意：

> 如果硬规则已经失败，judge 不能把它洗成通过。

## 4. 现在到底有哪些“任务类型”在测

当前主评测集大致在测四类任务：

### 4.1 计划类
例子：
- 阅读一个项目
- 输出一个优化计划
- 不允许改代码
- 回复里必须包含一些结构化内容

它主要测试：
- 是否理解任务
- 是否遵守“只计划不执行”
- 是否输出了要求的信息结构

### 4.2 Bugfix 类
例子：
- 修一个购物车负数数量 bug
- 要求运行 pytest
- 要求不要改无关文件

它主要测试：
- 是否真修了 bug
- 是否改在正确文件
- 是否有越界修改
- 是否完成了验证步骤

### 4.3 Test completion 类
例子：
- 只补测试
- 只能改测试文件
- 需要运行 pytest

它主要测试：
- 是否理解“只能改测试”这个边界
- 是否真的补上了测试
- 是否没有顺手去改产品代码

### 4.4 Scaffold 类
例子：
- 在空目录里初始化一个最小模块结构
- 最终要说明下一步

它主要测试：
- 面对空项目时是否能产出基本结构
- 是否给出合理的后续说明

所以现在这套体系测的不是单一能力，而是：

- 计划能力
- 修 bug 能力
- 写测试能力
- 初始化项目结构能力

---

## 5. 最关键的一部分：评测指标到底是什么

下面是这套体系里最重要的指标，以及它们**具体是怎么计算出来的**。

---

## 5.1 Pass / Fail / Error

这是最顶层结果。

### Pass
表示：
- 没有执行异常
- 所有硬规则都通过
- 如果有 judge，也没有被 judge 判成 fail

### Fail
表示：
- 虽然流程跑完了，但有硬规则失败
- 或者 judge 明确给了 fail（且没有硬规则先报错）

### Error
表示：
- 任务执行过程中直接抛异常，没能正常跑完

这个区分很重要：
- **error** 是“系统没跑完”
- **fail** 是“跑完了但不达标”

---

## 5.2 Tool call count

定义：

> 这个场景里，Agent 一共调用了多少次工具。

计算方式：
- 把整条执行轨迹里的所有 tool calls 数量加总。

用途：
- 衡量任务完成成本
- 配合 budget 判断是否“乱搜乱试”

例如：
- 一个简单测试补全任务如果调了 24 次工具，可能说明上下文命中率不高
- 一个 bugfix 只调了 5 次工具，可能更高效

---

## 5.3 Redundant tool call count / redundant tool rate

这是一个很关键的“工具抖动”指标。

### 定义
如果同一个工具，用**完全相同的参数**被重复调用多次，就记为冗余调用。

例如：
- 连续两次 `grep` 同一个关键词
- 多次读取同一个文件同一段内容
- 多次跑完全一样的 pytest 命令

### 计算方式
先给每次工具调用生成一个签名：

`工具名 + 参数序列化结果`

比如：
- `grep:{"pattern":"cart","path":"."}`
- `read_file:{"path":"shop/cart.py"}`

如果某个签名出现次数 > 1，就算作冗余签名。

当前结果里保留：
- 冗余调用签名列表
- 冗余调用数量

### Redundant tool rate
定义：

> 冗余调用数 / 总工具调用数

公式：

```text
redundant_tool_rate = redundant_tool_call_count / tool_call_count
```

### 这个指标在表达什么
它反映的是：

> Agent 有没有在重复做已经做过的事。

值越高，通常表示：
- 上下文利用差
- 搜索策略抖动
- 轨迹不够稳定

---

## 5.4 Check pass rate

定义：

> 场景要求执行的外部检查命令里，有多少比例通过了。

例如一个场景定义了 2 个 checks：
- pytest
- lint

如果只过了 1 个，那么：

```text
check_pass_rate = 1 / 2 = 0.5
```

如果场景没有 checks，当前实现里这个值记为 0。

### 它的重要性
这个指标直接告诉你：

> Agent 不只是“说自己做完了”，而是真的让外部验证通过了多少。

---

## 5.5 Requirement hit rate

这是当前最容易误解，但也很实用的指标。

### 它想回答的问题

> 这个任务要求的关键点，Agent 实际命中了多少？

### 当前计算口径
系统会把以下东西都当作“要求项”：

1. **required phrases**
   - 回复里必须出现的短语
2. **required changed files**
   - 必须改到的文件
3. **acceptance criteria**
   - 场景写出来的验收项（当前实现是看这些字符串有没有出现在最终回复里）
4. **checks**
   - 需要执行且通过的检查命令

### 计算方式
假设一个场景有：

- required phrase 2 项
- required changed file 1 项
- acceptance criteria 2 项
- checks 1 项

总要求项数：

```text
total = 2 + 1 + 2 + 1 = 6
```

如果实际命中了：
- 2 个 required phrases
- 1 个 required changed file
- 1 个 acceptance criterion
- 1 个 check

那么：

```text
matched = 2 + 1 + 1 + 1 = 5
requirement_hit_rate = 5 / 6 = 0.8333
```

### 它和 pass/fail 的区别
- pass/fail 是硬二值
- requirement hit rate 是连续值

所以它能区分：
- 完全没做对
- 做对一半
- 只差一点点

它特别适合比较不同策略的“接近正确程度”。

---

## 5.6 Self-correction rate

这个指标测的是：

> Agent 遇到工具错误以后，能不能自己继续操作并恢复回来。

### 当前实现里的“可恢复失败”定义
如果某一轮工具事件里出现 `error`，就记为一次 failure。

之后如果：
- Agent 继续调用工具
- 并进入后续正常推进

那就记为一次 recovered failure。

### 计算方式

```text
self_correction_rate = recovered_failures / recoverable_failures
```

### 举例
假设整个轨迹里：
- 一共出现了 4 次工具失败
- 其中 3 次失败后，Agent 继续行动并恢复推进

那么：

```text
self_correction_rate = 3 / 4 = 0.75
```

### 这个指标在看什么
它不是看“从没出错”，而是看：

> 出错以后会不会自己修回来。

这对 Agent 很重要，因为真实环境里：
- 命令可能失败
- 路径可能写错
- grep 可能没命中
- 测试第一次可能不通过

会自修的 Agent，比“一错就死”的 Agent 强很多。

---

## 5.7 Recovery cost

self-correction rate 告诉你“能不能修回来”，但还不够。

recovery cost 测的是：

> 修回来花了多少额外工具调用成本。

### 当前计算逻辑
一旦发生失败，系统开始累计“失败之后又多调了多少工具”。

如果后面恢复成功，就把这段额外调用数记入 recovery cost。

### 举例
假设：
- 第 5 轮发生失败
- 后面又调了 3 次工具才恢复成功

那么这次恢复的 cost = 3。

如果整个场景里出现两次恢复：
- 第一次 cost = 3
- 第二次 cost = 2

那么：

```text
recovery_cost = 5
```

### 这个指标有什么用
它区分了两种“都能修回来”的 Agent：

- A：失败后 1 次就纠正
- B：失败后反复试 6 次才纠正

二者 self-correction rate 可能都一样高，但 recovery cost 会不同。

---

## 5.8 Latency

定义：

> 这个场景从开始执行到结束执行，总共花了多少毫秒。

这是 wall-clock latency。

它受很多因素影响：
- 模型响应时间
- 工具调用次数
- 外部 checks 时间
- 任务复杂度

单场景记录的是 latency_ms。
整体汇总时通常看：
- p50（中位数）
- p90（当前实现近似取最大值）

### 它的用途
- 比较不同策略是否更慢
- 看某个功能是否让整体执行代价上升太多

---

## 5.9 Token usage

定义：

> 这个场景执行过程中累计消耗了多少 token。

当前是把 trace 里各轮 usage 相加。

整体汇总时也看：
- token p50
- token p90（当前实现近似取最大值）

### 它表达什么
- 成本
- 上下文策略是否膨胀
- 某个功能是否显著增加 prompt / response 开销

---

## 6. “硬规则判定”具体判什么

这部分决定一条场景为什么 pass/fail。

系统当前会检查：

### 6.1 不允许改文件
如果场景要求“不得修改文件”，但最终发生了任何文件改动，就失败。

### 6.2 必须包含的短语
如果要求回复里必须出现某些词，但最终回复没出现，就失败。

### 6.3 禁止出现的短语
如果要求不能出现某些词，但最终回复出现了，就失败。

### 6.4 工具调用预算
如果定义了最大工具调用数，而实际超过了，就失败。

### 6.5 文件改动预算
如果定义了最大改动文件数，而实际超过了，就失败。

### 6.6 必须修改的文件
如果某些文件本来必须改，但实际没改到，就失败。

### 6.7 禁止修改的文件
如果某些文件不允许改，但实际改了，就失败。

### 6.8 允许修改范围
如果场景规定“只能改这些文件”，而实际还改了别的文件，就失败。

### 6.9 外部检查失败
如果 scenario 里定义的 checks 有任意一个没通过，就失败。

所以可以把 deterministic checks 理解成：

> **任务边界守卫器**

它负责回答：
- 有没有做该做的
- 有没有做不该做的
- 有没有超预算
- 有没有通过验证

## 7. 用两个具体例子解释“测评到底怎么工作”

## 7.1 例子一：bugfix.cart_negative_quantity.001

任务大意：
- 修复购物车对负数数量缺少校验的问题
- 运行 `pytest tests/test_cart.py`

这个场景真正测的是：

### 输入
- 起点：同一份商店项目
- 用户要求：修 bug + 跑 pytest

### 约束
- 必须改到 `shop/cart.py`
- 禁止改 `README.md`
- 必须执行 `pytest tests/test_cart.py`
- 最多 30 次工具调用
- 最多改 3 个文件

### 评测关注点
1. 你有没有真的改到 cart 逻辑
2. 你有没有误改 README 这种无关文件
3. 你有没有跑 pytest
4. pytest 最终有没有通过
5. 你是否在预算内完成
6. 你的过程是不是比较稳定

### 假设一个 run 的结果
- 改了：`shop/cart.py`
- 没改：`README.md`
- pytest 通过
- 工具调用 12 次
- 其中 2 个是重复调用
- 最终回复提到了“负数数量”和“pytest tests/test_cart.py`

那么：
- deterministic：通过
- status：大概率 pass
- tool_call_count = 12
- redundant_tool_call_count = 2
- redundant_tool_rate = 2 / 12 = 0.167
- check_pass_rate = 1 / 1 = 1.0

如果该场景要求项共有 4 个：
- required changed file 1
- acceptance criteria 2
- check 1

且全部命中，则：

```text
requirement_hit_rate = 4 / 4 = 1.0
```

如果中途有一次工具失败，但随后继续推进并恢复：
- recoverable_failures = 1
- recovered_failures = 1
- self_correction_rate = 1.0
- recovery_cost 取决于恢复前多调了多少工具

---

## 7.2 例子二：tests.cart_regression.001

任务大意：
- 只补测试
- 只能改测试文件
- 运行 pytest

这个场景测的重点和 bugfix 不同，它主要在测：

1. 是否理解“只改测试文件”的边界
2. 会不会忍不住去改产品代码
3. 是否能补出有效测试并跑通

### 输入
- 起点：同一份商店项目
- 用户要求：补边界测试

### 约束
- 只允许改 `tests/test_cart.py`
- 必须改 `tests/test_cart.py`
- 必须跑 pytest
- 最多 24 次工具调用
- 最多改 1 个文件

### 一个典型失败例子
如果 Agent 同时改了：
- `tests/test_cart.py`
- `shop/cart.py`

那么即使 pytest 通过，也会失败。

因为它违反了：
- `allowed_changed_files`
- `max_changed_files`

这正说明这套评测不是“只看功能是否过”，而是也看：

> **是否按用户要求的协作边界完成。**

---

## 8. 汇总报表里到底在看什么

单个场景跑完以后会得到一个结果；多个场景组成一个 suite 后，会生成汇总指标。

当前汇总主要看这些：

### 8.1 通过情况
- total
- passed
- failed
- errors
- pass_rate

其中：

```text
pass_rate = passed / total
```

### 8.2 性能和成本
- latency_p50_ms
- latency_p90_ms
- token_p50
- token_p90

### 8.3 工具行为质量
- tool_call_p50
- redundant_tool_rate_p50

### 8.4 任务完成度
- requirement_hit_rate_p50
- check_pass_rate

### 8.5 出错恢复能力
- self_correction_rate_p50
- recovery_cost_p50

这里的 p50 可以理解成：
- 把所有 scenario 的该指标拿出来
- 取中位数

这样不会被单个极端场景带偏太多。

## 9. LLM judge 在这里到底起什么作用

很多人会以为 judge 是“最终裁判”，但在这里它不是。

它更像：

> **一个补充性的语义评审器**

它看的是：
- 原始任务
- 验收条件
- 改了哪些文件
- checks 是否通过
- 最终回复写了什么

然后输出：
- score（0~1）
- passed / failed
- verdict
- reasoning

### 为什么需要 judge
因为有些东西很难靠硬规则精确表达，例如：
- 计划是不是“像样”
- 回复有没有真正覆盖要求
- 改动是否体现了目标意图

### 但它不能推翻硬规则
如果 deterministic checks 已经失败，例如：
- 改错文件
- pytest 没过
- 超预算

那 judge 就算主观上觉得“还行”，最终也不能把结果变成 pass。

这点非常重要。

## 10. 这套体系最适合拿来比较什么

它最适合做的是：

> 比较两个版本的 GhostBot，在同一批固定任务上的整体表现差异。

例如比较：
- baseline：旧上下文策略
- candidate：新 graph-aware context 策略

两边跑同一套场景后，你可以直接比较：

### 结果层面
- pass 数有没有增加
- fail 数有没有减少

### 成本层面
- latency p50 是否上升
- token p50 是否上升

### 轨迹质量层面
- redundant tool rate 是否下降
- self-correction rate 是否提升

### judge 层面
- judge pass 数是否更多
- judge score 中位数是否更高

所以它不是“看一次 run 好不好看”，而是：

> **用同一起点、同一任务集合，做版本间可重复比较。**

## 11. 现在这套测评体系的边界

虽然已经能用，但当前它有几个要清楚的边界。

### 11.1 它主要测“任务执行”
它现在最擅长回答的是：
- 这个 Agent 能不能完成 bugfix / 测试补全 / scaffold / plan 任务

但对于我们最近在做的能力，例如：
- graph recall 准不准
- related test recall 是否提升
- context packing 是否更优
- memory 注入是否真正有用

它还没有单独拆出非常精准的专项指标。

### 11.2 Requirement hit rate 目前偏“命中字串/命中项”
它是实用的，但还不算完美。

例如 acceptance criteria 目前本质上还是通过“最终回复里是否出现相应字符串”来衡量。

这意味着：
- 它适合做粗粒度完成度估计
- 但还不等于真正语义级验收

### 11.3 冗余工具调用判定比较保守
只有“工具名 + 参数完全相同”的重复，才算冗余。

这会漏掉一些“语义上重复、参数略不同”的抖动。

例如：
- 两次 grep 模式稍微不同，但本质都在盲搜同一块内容

当前还抓不到。

### 11.4 Self-correction 也是一个近似指标
它是按“先出现错误，再继续调用工具并恢复推进”来估计的。

能抓住大方向，但还不是精细的失败恢复建模。

## 12. 如果你把它看成一个实验系统，最核心的输入/输出分别是什么

### 输入
1. 一组固定起点项目
2. 一组固定任务描述
3. 每个任务的边界约束
4. 每个任务的验证命令
5. 同一个 GhostBot 版本或策略版本

### 输出
1. 每个任务是否 pass/fail/error
2. 改动文件范围是否正确
3. 外部验证是否通过
4. token / latency 成本
5. 工具调用总数与冗余率
6. requirement hit rate
7. self-correction rate / recovery cost
8. 可选 judge score
9. 两个版本之间的增量对比

所以你可以把它抽象成一句话：

> **同一起点项目 + 同一组用户任务 + 同一套硬约束与验证命令 → 产出结果正确性、过程质量、执行成本三大类指标。**

## 13. 最后给一个最简心智模型

如果你想最快理解这套评测体系，可以只记这三层：

### 第一层：任务有没有完成
看：
- pass / fail / error
- checks 是否通过
- 是否改对文件

### 第二层：任务是不是按要求完成
看：
- 有没有越界改动
- 有没有超工具预算
- requirement hit rate 多高

### 第三层：完成过程好不好
看：
- latency
- tokens
- redundant tool rate
- self-correction rate
- recovery cost

所以它不是只问：
- “会不会做”

而是同时问：
- “有没有做对”
- “有没有按边界做”
- “做得是否高效稳定”

这才是 GhostBot 现在这套测评体系真正测的东西。
